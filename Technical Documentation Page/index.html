Introduction

Most developers interact with Git through high-level commands like git commit, git push, and git merge. But beneath these familiar porcelain commands lies a powerful content-addressable filesystem — one that's surprisingly simple once you understand its core model.

Git stores everything in a directory called .git/ at the root of your repository. Inside it, you'll find objects, references, configuration, and more. Understanding this structure demystifies almost every confusing Git behaviour you've encountered.

Why learn internals?

When a rebase goes wrong or a merge produces unexpected conflicts, understanding the object model lets you inspect and repair state directly — instead of guessing with high-level commands.

Git Object Model

At its core, Git is a key-value store. Every piece of content is hashed with SHA-1 (transitioning to SHA-256 in newer versions), and stored as an object. There are exactly four object types:

Type Stores Contains
blob File content Raw bytes of a file at a point in time
tree Directory listing Pointers to blobs and other trees
commit Snapshot metadata Tree pointer, parent, author, message
tag Named reference Points to any object with extra metadata

All objects are stored under .git/objects/. The first two hex characters of the hash become a subdirectory; the remaining 38 form the filename.

.git/objects/ filesystem 
.git/objects/
├── 4b/
│   └── 825dc642cb6eb9a060e54bf8d69288fbee4904  # empty tree
├── a1/
│   └── f3c9e2b84d0e67f21ac98302de4a5e1c70b3d1  # a blob
└── pack/
    └── *.pack                                  # packed objects

Blobs & Trees

A blob stores the raw contents of a file — no filename, no permissions, just bytes. Two files with identical contents will always share the same blob hash. This is how Git deduplicates data.

You can create a blob manually using the git hash-object plumbing command:

Bash 
$ echo "Hello, Git" | git hash-object --stdin -w
8ab686eafeb1f44702738c8b0f24f2567c36da6d

$ git cat-file -t 8ab686ea
blob

$ git cat-file -p 8ab686ea
Hello, Git

A tree represents a directory. Each entry has a mode, a type, a hash, and a name. The mode 100644 indicates a regular file, while 040000 indicates a subdirectory (another tree).

tree object contents output 
100644 blob a1f3c9e2b84d...    README.md
100644 blob 8ab686eafeb1...    main.go
040000 tree c3d4e5f60718...    src/

Commits

A commit object ties everything together. It points to a single root tree, lists zero or more parent commits, and stores author metadata. The commit hash changes if any of these fields change — which is why amending a commit produces a new hash.

commit abc123
tree f7a3b1...
parent e91d44...
author Ana M.
date 2026-04-18
tree f7a3b1
blob a1f3c9 README
blob 8ab686 main.go
tree c3d4e5 src/
blob a1f3c9
raw file bytes...

The raw commit object can be inspected with git cat-file -p HEAD, which prints the tree hash, parent hashes, author/committer lines, and commit message.

raw commit object
git
tree   f7a3b1d8e29c4a5601fe8d72b3c94f015a8e7bcd
parent e91d449cba2f3c78a1d06e5b48f9320c1a7e4583
author     Ana Martinez <ana@example.com> 1713398400 +0700
committer  Ana Martinez <ana@example.com> 1713398400 +0700
 
Add initial project structure

References

Hashes are unwieldy. Git solves this with references — named pointers stored as plain text files inside .git/refs/. A branch is simply a file containing a 40-character commit hash.

Bash 
$ cat .git/refs/heads/main
abc123f4e5d6c7b8a9012345678901234567890a

$ cat .git/HEAD
ref: refs/heads/main

The special reference HEAD usually points to a branch (attached state). When you check out a specific commit hash, HEAD stores the hash directly — this is the infamous "detached HEAD" state.

Symbolic refs

References like MERGE_HEAD, ORIG_HEAD, and CHERRY_PICK_HEAD are temporary refs Git creates during in-progress operations. You can inspect them the same way — they're just files.

Plumbing Commands

Git distinguishes between porcelain (user-facing) and plumbing (low-level) commands. Plumbing commands operate directly on objects and refs — they're the building blocks that porcelain commands use internally.

Tip

You can chain plumbing commands to walk the full object graph: git rev-parse HEAD → git cat-file -p <hash> → git ls-tree <tree-hash>.

Command What it does
git cat-file Inspect object type, size, or content
git hash-object Compute hash and optionally write an object
git ls-tree List contents of a tree object
git rev-parse Resolve refs and expressions to hashes
git update-ref Safely update a reference file
git write-tree Create a tree object from the index

Packfiles

Storing every version of every file as a separate loose object would be wasteful. Git periodically runs garbage collection (git gc) to combine loose objects into binary packfiles, stored under .git/objects/pack/.

Each packfile (.pack) is accompanied by an index (.idx) for fast random access. Within the pack, objects are delta-compressed against similar objects — a one-line change to a 10,000-line file stores only the delta, not the full content.

inspect a packfile
bash
# List objects in a packfile
git verify-pack -v .git/objects/pack/*.idx
 
# Repack with aggressive delta compression
git repack -a -d --depth=50 --window=250
 
# Show packfile statistics
git count-objects -vH

When you clone a repository, Git sends exactly one packfile over the wire — containing all the objects needed to reconstruct every requested ref. This is why git clone is a single network round-trip.

Why It Matters

  • Fearless debugging When a rebase, merge, or reset goes wrong, you can drop down to git cat-file and git ls-tree to inspect the raw object graph — instead of blindly running git reflog and hoping for the best.
  • Confident data recovery Because Git almost never deletes objects immediately, understanding loose objects and packfiles means you can recover "lost" commits, blobs, and branches long after they disappear from the visible history.
  • Faster repository maintenance Knowing when and why to run git gc or git repack lets you keep repository size under control — especially important for large repos with binary assets or long histories.
  • Cleaner mental model of branching Once you see that a branch is just a file with a hash inside .git/refs/heads/, branching and merging stop being magic. You understand exactly what git switch, git reset, and git cherry-pick are actually doing to refs and the object store.
  • Better tooling and automation Plumbing commands are stable, scriptable, and designed for machine consumption. Building CI scripts, custom hooks, or repository analysis tools becomes far more reliable when you use git rev-parse and git for-each-ref instead of parsing porcelain output.
